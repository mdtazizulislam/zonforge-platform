# ZONFORGE-STRIPE-01

## 1. Current Billing Ownership Model
Current schema has `users`, `tenants (user_id)`, and `subscriptions (user_id)`.
Billing ownership selected: **A. subscription belongs to user**.
Reason: existing auth token and ownership are user-centric, and existing subscription table already references `user_id`.

## 2. Files Changed
- apps/backend/src/stripe.ts
- apps/backend/src/index.ts
- apps/backend/src/db.ts
- apps/backend/.env.example

## 3. DB Changes
Added additive billing persistence and idempotency tables in backend init:
- `billing_subscriptions`
  - id
  - owner_user_id
  - stripe_customer_id
  - stripe_subscription_id
  - stripe_checkout_session_id
  - stripe_price_id
  - subscription_status
  - current_period_start
  - current_period_end
  - cancel_at_period_end
  - last_webhook_event_id
  - created_at
  - updated_at
- `billing_webhook_events`
  - event_id (PK)
  - event_type
  - created_at

Indexes:
- unique owner index on `billing_subscriptions(owner_user_id)`
- unique partial indexes for stripe customer/subscription/checkout session IDs

Rollback note:
- additive-only schema; rollback can drop `billing_webhook_events` and `billing_subscriptions` if needed.

## 4. API Endpoints Added/Updated
- `POST /billing/checkout-session`
  - auth required
  - creates/uses Stripe customer
  - creates Stripe subscription checkout session
  - returns `{ sessionId, url }`
- `POST /billing/webhook`
  - raw body signature verification with `STRIPE_WEBHOOK_SECRET`
  - idempotent event handling via `billing_webhook_events`
- `GET /billing/status`
  - auth required
  - returns persisted billing row for authenticated user
- Legacy compatibility maintained:
  - `POST /webhook/stripe` routes through same webhook handler

Handled events:
- checkout.session.completed
- customer.subscription.created
- customer.subscription.updated
- customer.subscription.deleted
- invoice.paid
- invoice.payment_failed
